## Περίληψη

Το παρόν έγγραφο παρουσιάζει μια ολοκληρωμένη τεχνική επισκόπηση του Συστήματος Διαχείρισης Φοιτητών (SMS), μιας πλήρως λειτουργικής, δίγλωσσης (Ελληνικά/Αγγλικά) πλατφόρμας ακαδημαϊκής διαχείρισης για τον τομέα της μεταδευτεροβάθμιας τεχνικής εκπαίδευσης. Το σύστημα ενσωματώνει ένα backend FastAPI σε Python με ένα frontend React/TypeScript, διατίθεται ως αυτόνομο πρόγραμμα εγκατάστασης Windows και υποστηρίζει τέσσερις διαφορετικές λειτουργίες ανάπτυξης. Διαθέτει μηχανισμό σταθμισμένης βαθμολόγησης πολλαπλών συνιστωσών, προγνωστική ανάλυση με μοντελοποίηση τάσεων μέσω γραμμικής παλινδρόμησης, προσαρμόσιμες αναφορές dashboard, λεπτομερή έλεγχο πρόσβασης βάσει ρόλων, καθώς και αγωγό CI/CD που охватει 38 ροές εργασίας GitHub Actions. Το έγγραφο απευθύνεται σε κοινό ακαδημαϊκών επιστημόνων υπολογιστών και επαγγελματιών, και παρέχει αρχιτεκτονικά διαγράμματα, αλγοριθμικές περιγραφές και αναπαραγώγιμα τεκμήρια δοκιμών για όλα τα κύρια υποσυστήματα.

## 1. Πλαίσιο και Κίνητρο Έργου

Τα ΜΙΕΕΚ (Μεταλυκειακά Ινστιτούτα Επαγγελματικής Εκπαίδευσης και Κατάρτισης) στην Κύπρο λειτουργούν υπό νομοθετικό πλαίσιο που επιβάλλει δομημένη βαθμολόγηση, τακτική παρακολούθηση παρουσίας και διαφανείς ακαδημαϊκές αναφορές. Πριν από το SMS, τα μέλη του διδακτικού προσωπικού στο τμήμα της Λεμεσού διαχειρίζονταν τα αρχεία φοιτητών σε απομονωμένα υπολογιστικά φύλλα και έντυπα μητρώα. Αυτό δημιουργούσε κενά στο ίχνος ελέγχου, απέτρεπε τη συγκεντρωτική ανάλυση δεδομένων μεταξύ μαθημάτων και παρήγαγε ασυνεπείς εκδόσεις βαθμολογίων.

Το SMS σχεδιάστηκε από μηδενική βάση για να αντιμετωπίσει αυτές τις ανάγκες. Τρεις σχεδιαστικοί περιορισμοί διαμόρφωσαν όλες τις αρχιτεκτονικές αποφάσεις:

1. **Καμία εξάρτηση από κοινή υποδομή** — Το ίδρυμα δεν διαθέτει διακομιστή με συνεχή λειτουργία. Το σύστημα πρέπει να εκτελείται στον φορητό υπολογιστή Windows ενός εκπαιδευτικού χωρίς Docker αν χρειαστεί.
2. **Πλήρης διγλωσσία** — Όλα τα κείμενα, τα μηνύματα σφάλματος, οι εξαγωγές PDF και οι ετικέτες διεπαφής πρέπει να είναι διαθέσιμα ταυτόχρονα στα Αγγλικά και στα Νέα Ελληνικά.
3. **Συμφωνία θεσμικής βαθμολόγησης** — Το σύστημα πρέπει να υλοποιεί πιστά και ιχνηλάσιμα την τυπική κλίμακα βαθμολόγησης της Κύπρου (0–20 μονάδες, κατώφλι επιτυχίας 10, με αντιστοίχιση GPA).

### 4.3 Λεπτομέρεια Μοντέλου Βαθμού

```python
class Grade(SoftDeleteMixin, Base):
    __tablename__ = "grades"

    id              = Column(Integer, primary_key=True)
    student_id      = Column(Integer, ForeignKey("students.id"), index=True)
    course_id       = Column(Integer, ForeignKey("courses.id"), index=True)
    assignment_name = Column(String(200), nullable=False)
    category        = Column(String(100), index=True)   # αντιστοιχεί σε κανόνα αξιολόγησης
    grade           = Column(Float, nullable=False)
    max_grade       = Column(Float, default=100.0)
    weight          = Column(Float, default=1.0)        # ενδοκατηγορικό βάρος
    date_assigned   = Column(Date, index=True)
    date_submitted  = Column(Date, index=True)
    notes           = Column(Text)

    @property
    def percentage(self) -> float:
        return (grade / max_grade) * 100 if max_grade > 0 else 0.0
```

## 5. Βασικό Χαρακτηριστικό: Μηχανισμός Βαθμολόγησης

### 5.1 Εννοιολογικός Σχεδιασμός

Το σύστημα βαθμολόγησης υλοποιεί ένα **τριεπίπεδο ιεραρχικό μοντέλο**:

```
Μάθημα
 └── Κανόνες Αξιολόγησης  (στήλη JSON στο Μάθημα)
      └── Κατηγορία  (π.χ., "Ενδιάμεσες Εξετάσεις", "Εργασίες", "Συμμετοχή")
           ├── βάρος_κατηγορίας  (ποσοστό τελικού βαθμού, 0–100)
           ├── πολλαπλασιαστής   (float — βαρύτητα βαθμών ημερήσιας επίδοσης)
           └── is_attendance     (bool — οδηγεί υπολογισμό βάσει παρουσίας)
```

### 5.2 Αλγόριθμος Τελικού Βαθμού

Η μέθοδος `AnalyticsService.calculate_final_grade()` (`src/backend/services/analytics_service.py`) υπολογίζει τον τελικό βαθμό ως εξής:

**Βήμα 1 — Ομαδοποίηση εισόδων ανά κατηγορία**

```python
for rule in evaluation_rules:
    category_name   = rule["name"]
    category_weight = rule["weight"]  # π.χ. 0.40 για 40%
    is_attendance   = rule.get("is_attendance", False)
    multiplier      = rule.get("multiplier", 1.0)
```

**Βήμα 2 — Υπολογισμός βαθμού ανά κατηγορία**

*Για κατηγορίες βάσει βαθμών:*
```
σταθμισμένο_άθροισμα = Σ (βαθμός_i / max_βαθμός_i) × ενδοβάρος_i
                      + Σ (ημ_επίδοση_j / max_βαθμός_j) × πολλαπλασιαστής_j
συνολικό_βάρος      = Σ ενδοβάρος_i + Σ πολλαπλασιαστής_j
βαθμός_κατηγορίας   = (σταθμισμένο_άθροισμα / συνολικό_βάρος) × 100
```

*Για κατηγορίες βάσει παρουσίας:*
```
βαθμός_κατηγορίας = (αριθμός_παρόντων / σύνολο_συνεδριών) × 100
```

**Βήμα 3 — Εφαρμογή ποινής απουσίας**

```
ακατέργαστος_τελικός = Σ (βαθμός_κατηγορίας_k × βάρος_κατηγορίας_k)
ποινή               = course.absence_penalty × αδικαιολόγητες_απουσίες
τελικός_βαθμός      = max(0.0, ακατέργαστος_τελικός − ποινή)
```

**Βήμα 4 — Μετατροπή σε πολλαπλές κλίμακες**

```python
gpa_4    = (τελικός_βαθμός / 100) × 4.0       # Αμερικανική κλίμακα GPA
ελλ_20   = (τελικός_βαθμός / 100) × 20.0      # Κυπριακή/Ελληνική κλίμακα
γράμμα   = _βαθμός_σε_γράμμα(τελικός_βαθμός) # Α+/Α/Β+/Β/Γ/Δ/Αποτ.
```

### 6.2 Υπηρεσία Προγνωστικής Ανάλυσης

Το `PredictiveAnalyticsService` (`src/backend/services/predictive_analytics_service.py`) υλοποιεί ένα στατιστικό επίπεδο μοντελοποίησης σε **καθαρή Python** — χωρίς εξωτερικά πλαίσια ML. Αυτό επιτρέπει την ανάπτυξη σε απομονωμένα συστήματα και διατηρεί τις προβλέψεις διαφανείς και ελέγξιμες.

#### 6.2.1 Πρόβλεψη Τάσης Βαθμού — Γραμμική Παλινδρόμηση Ελαχίστων Τετραγώνων (OLS)

```python
def predict_grade_trend(
    historical_grades: List[Tuple[datetime, float]],
    weeks_ahead: int = 4
) -> Dict[str, Any]:
```

**Αλγόριθμος:**

Δεδομένης χρονοσειράς `(t₁, g₁), (t₂, g₂), …, (tₙ, gₙ)` όπου `tᵢ` είναι χρονοσφραγίδα POSIX και `gᵢ ∈ [0, 100]`:

```
Κλίση OLS:     β = Σ(tᵢ − t̄)(gᵢ − ḡ) / Σ(tᵢ − t̄)²
Τεταγμένη OLS: α = ḡ − β × t̄
Πρόβλεψη:      ĝ(t) = α + β × t   (περιορισμένη σε [0, 100])
```

Για κάθε μία από τις `weeks_ahead` μελλοντικές εβδομάδες η μέθοδος επιστρέφει:
- `predicted_grade` — πρόβλεψη OLS, περιορισμένη σε [0, 100]
- `confidence` — αντίστροφη κανονικοποιημένη τυπική απόκλιση ιστορικών βαθμών
- `current_trend` — `"improving"` (β > 1/εβδομάδα), `"declining"` (β < −1/εβδομάδα), ή `"stable"`
- `r_squared` — συντελεστής προσδιορισμού

Απαιτείται ελάχιστο **3 σημεία δεδομένων**. Κλήσεις με λιγότερα επιστρέφουν `"insufficient_data"` χωρίς εξαίρεση.

#### 6.2.2 Πρόβλεψη Μοτίβου Παρουσίας

```python
def predict_attendance_pattern(
    attendance_records: List[Tuple[datetime, bool]]
) -> Dict[str, Any]:
```

Η μέθοδος:
1. Χωρίζει εγγραφές ανά ημέρα εβδομάδας (7 ομάδες)
2. Υπολογίζει ποσοστό παρουσίας ανά ημέρα: `παρόντες / σύνολο_συνεδριών`
3. Εντοπίζει την ημέρα υψηλότερου κινδύνου (χαμηλότερο ποσοστό)
4. Κατατάσσει τον φοιτητή ως `at_risk` όταν το συνολικό ποσοστό < 0.75
5. Εκτιμά προβλεπόμενες απουσίες για τις υπόλοιπες συνεδρίες εξαμήνου

#### 6.2.3 Βαθμολόγηση Κινδύνου Φοιτητή

```
βάση = 50 (ουδέτερο)

Συνιστώσα βαθμού:
  declining → +20   stable → 0   improving → −10

Συνιστώσα παρουσίας:
  at_risk → +20   ποσοστό < 0.9 → +10

Συνιστώσα δέσμευσης:
  χαμηλό_ποσοστό_υποβολής → +10
```

Τελική κατηγοριοποίηση:
- 0–25: `χαμηλός_κίνδυνος` (πράσινο)
- 26–60: `μέτριος_κίνδυνος` (πορτοκαλί)
- 61–100: `υψηλός_κίνδυνος` (κόκκινο)

## 14. Στάση Ασφαλείας

### 14.1 Αυθεντικοποίηση

- Κατακερματισμός κωδικού: `passlib[bcrypt]`
- Διακριτικά JWT (PyJWT 2.13.0)
- Περιορισμός ρυθμού σύνδεσης: διαμορφώσιμο ανά IP
- Προστασία CSRF μέσω πολιτικής SameSite cookie

### 14.2 Ασφάλεια Εξαρτήσεων

Το Dependabot σαρώνει τρία αρχεία manifest συνεχώς. Το v1.18.30 επέλυσε **25 ειδοποιήσεις Dependabot**, συμπεριλαμβανομένων CVE στο starlette, κρυπτογραφία, python-multipart και PyJWT. **Τρέχουσα κατάσταση: 0 ανοιχτές ειδοποιήσεις Dependabot · 0 ανοιχτές ειδοποιήσεις CodeQL.**

### 14.3 Επικύρωση Εισόδου

- Όλες οι είσοδοι API επικυρώνονται από Pydantic v2 με αυστηρούς τύπους
- Προστασία διάσχισης μονοπατιών στα endpoints αντιγράφων ασφαλείας
- Ασφαλή μοτίβα regex (50 ευπάθειες ReDoS επιλύθηκαν στο v1.18.26)
- Απαντήσεις σφαλμάτων παραγωγής ποτέ δεν εκθέτουν ίχνη Python stack

### 14.4 Ασφάλεια Αντιγράφων

- Κρυπτογραφημένη λειτουργία αντιγράφου (AES μέσω `cryptography.Fernet`)
- Παράλληλο αρχείο `.sha256` για κάθε αντίγραφο
- Επαλήθευση gzip πριν από αποθήκευση
- Κλείσιμο παλαιών αντιγράφων μέσω `BACKUP_KEEP_COUNT`

## 15. Σύνοψη Διακριτικών Χαρακτηριστικών

### 15.1 Μηχανισμός Βαθμολόγησης Θεσμικής Ποιότητας

Σε αντίθεση με συστήματα που αποθηκεύουν έναν μόνο αριθμητικό βαθμό ανά εργασία, το SMS υλοποιεί ολόκληρη τη Κυπριακή βαθμολογική σύμβαση επαγγελματικής εκπαίδευσης:
- Διαμορφώσιμοι ανά μάθημα πολυ-συνιστωσικοί κανόνες αξιολόγησης
- Σταθμισμένοι μέσοι όροι κατηγοριών με πολλαπλασιαστές ημερήσιας επίδοσης
- Αφαίρεση ποινής απουσίας με κατώτατο όριο στο 0
- Ταυτόχρονη έξοδος σε ποσοστό, GPA (4.0), Ελληνική (20 μονάδες) και βαθμό γράμματος
- Πλήρης ανάλυση κατηγορίας σε κάθε απόκριση βαθμού

### 15.2 Διγλωσσία από Σχεδιασμό

Η υποστήριξη Ελληνικών/Αγγλικών δεν είναι ένα στρώμα μετάφρασης που εφαρμόζεται εκ των υστέρων — επιβάλλεται σε κάθε επίπεδο: επικύρωση σχήματος, εξαγωγή PDF/Excel, όλο το κείμενο διεπαφής, ακόμα και μηνύματα σφάλματος.

### 15.3 Ανάπτυξη Μηδενικής Υποδομής (Lite Edition)

Η Lite Edition συμπτύσσει ολόκληρη τη στοίβα παραγωγής σε ένα μόνο αυτόνομο εκτελέσιμο Windows 97 MB χωρίς προαπαιτούμενα χρόνου εκτέλεσης. Τεχνικά μη τετριμμένο: το PyInstaller πρέπει να ενσωματώσει CPython, όλες τις εγγενείς βιβλιοθήκες, το προκατασκευασμένο frontend bundle και τα αρχεία γραμματοσειρών.

### 15.4 Προγνωστική Ανάλυση χωρίς Εξωτερικές Εξαρτήσεις ML

Οι προβλέψεις τάσης βαθμού και μοτίβου παρουσίας υλοποιούνται σε καθαρή Python χρησιμοποιώντας τυπικές βιβλιοθήκες, επιτρέποντας ανάπτυξη σε απομονωμένα συστήματα χωρίς PyTorch, scikit-learn ή οποιοδήποτε χρόνο εκτέλεσης ML.

### 15.5 Πλήρης Ίχνος Ελέγχου

Κάθε εγγραφή εγγραφής (δημιουργία βαθμού, σήμανση παρουσίας, ενημέρωση φοιτητή, αλλαγή ρόλου) δημιουργεί εγγραφή `AuditLog` με JSON snapshots `old_values` και `new_values`. Αυτό ικανοποιεί τις απαιτήσεις ελέγχου εκπαιδευτικού ιδρύματος της Κύπρου.

### 15.6 Offline-First για Αίθουσες Διδασκαλίας

Η αρχιτεκτονική ουράς offline (IndexedDB + συμφιλίωση WebSocket) αναγνωρίζει την πραγματικότητα της συνδεσιμότητας αίθουσας. Η σήμανση παρουσίας και η υποβολή βαθμών επιβιώνουν σε διακοπές δικτύου και συγχρονίζονται αυτόματα κατά την επανασύνδεση.

### 15.7 Παράδοση Τεσσάρων Πλατφορμών από Μία Κωδικοβάση

Το ίδιο FastAPI backend και React frontend εκτελείται ως:
1. Ζεύγος εγγενών διεργασιών Python + Node.js προγραμματιστή
2. Στοίβα Docker Compose παραγωγής
3. Αυτόνομο Windows EXE (Lite)
4. Android APK (Capacitor)

Όλα τέσσερα μοιράζονται ταυτόσημη λογική αυθεντικοποίησης, RBAC, βαθμολόγησης και αναλυτικών.

### 15.8 Κουλτούρα Ανάπτυξης Βάσει Τεκμηρίων
